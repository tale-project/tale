# `@tale/pii`

Multilingual PII detection, masking, and scrubbing for the Tale platform.

The library detects personally identifiable information (email addresses,
phone numbers, credit cards, IBANs, IP addresses, dates of birth, postal
addresses, national identity numbers, and more) in free-form text and either
masks it with replacement tokens (`[EMAIL]`, `[PHONE]`, …) or blocks the
input outright. Detection covers 50 locales spanning Latin, Cyrillic, Greek,
Arabic, Hebrew, CJK, and Indic scripts.

## Design principles

1. **Offline-only.** The library never opens a socket. Every detector is a
   regex, a checksum validator, or a call into bundled `libphonenumber-js`
   metadata. CI runs offline; the dataset-driven test fixtures are
   committed; the fixture generator reads only files inside the repo.
2. **Browser-safe.** Pure regex + string ops + Web Crypto where needed.
   No Node-only APIs in the core library; consumable from Convex V8, edge
   workers, and the browser.
3. **Never throws.** `scrubPii` and `createScrubber().scrub()` always
   return a `FilterOutcome` (`pass` / `modified` / `blocked`). Validator
   exceptions are caught and logged with the pattern name only — never the
   matched text (GDPR).
4. **ReDoS-bounded.** Every regex runs under `execWithBudget` (50 ms wall
   clock per pattern). Input is clamped to 50 KB before scanning. Custom
   user-supplied regexes are statically analyzed by `safe-regex2` at the
   Zod-schema layer before they ever reach the detector.
5. **Locale-aware.** Address and national-ID detection compose per-locale
   bundles; consumers opt in to the locales they care about, keeping the
   active pattern set small and the false-positive rate low.

## Quick start

```ts
import { createScrubber } from '@tale/pii';

const scrubber = createScrubber({
  mode: 'mask',
  patterns: {
    email: true,
    phone: true,
    creditCard: true,
    iban: true,
    address: { locales: ['en', 'de', 'fr'] },
  },
});

scrubber.scrub('Reach me at john@example.com or +1 (415) 555-0123.');
// { outcome: 'modified', text: 'Reach me at [EMAIL] or [PHONE].', … }
```

## Architecture

```
src/
├── core/        — outcome types, NFC normalization, ReDoS-safe regex exec
├── engine/      — detector, masker, Scrubber instance, PatternRegistry
├── patterns/    — built-in pattern implementations (email, phone, …)
│   ├── address/locales/    — 50 locale-specific address pattern bundles
│   └── national-ids/       — country-specific ID detectors + checksums
├── locales/     — per-locale config (keyword sets, postcode regex)
├── schemas/     — Zod schemas for runtime config validation
└── adapters/    — legacy compatibility shims
test/
├── fixtures/    — generated JSON test data (one dir per locale)
└── *.test.ts    — data-driven, ReDoS, Unicode, throughput suites
```

See [src/patterns/address/locales/README.md](./src/patterns/address/locales/README.md)
for the recipe to add a new locale.

## Test fixtures

Fixtures are generated by [`@tale/pii-fixtures`](../../tools/pii-fixtures/)
from committed dataset snapshots and frozen as JSON in
[`test/fixtures/<locale>/`](./test/fixtures/). Each locale carries 2,000+
test cases — a mix of positives (PII the detector must find) and negatives
(prose the detector must leave alone). Regenerate with:

```bash
bun run --filter @tale/pii-fixtures gen
```

The generator runs fully offline. Refreshing the underlying dataset snapshots
is a separate, manual maintainer workflow — see
[tools/pii-fixtures/refresh/README.md](../../tools/pii-fixtures/refresh/README.md).
